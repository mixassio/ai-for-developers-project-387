# Интеграционные сценарии (Playwright)

Интеграционные (E2E) тесты проверяют, что фронтенд (`frontend/`) и API работают
вместе в основных пользовательских сценариях сервиса «Календарь звонков».

## Что и против чего тестируем

- **UI:** React + Vite SPA на `http://localhost:5173`.
- **API:** реальный бэкенд `backend/` (Fastify, in-memory, **stateful**) на
  `http://localhost:8080`. Состояние хранится в памяти и сбрасывается при
  перезапуске процесса, поэтому каждый прогон Playwright стартует свежий бэкенд
  и начинает с чистого состояния (засеяны 3 типа событий: 15/30/60 минут).
- Оба сервера поднимает сам Playwright через `webServer` (см.
  `playwright.config.ts`). Фронтенд получает `VITE_API_URL=http://localhost:8080`
  и ходит на бэкенд напрямую (CORS на бэкенде разрешён).

Слоты детерминированы: будни 09:00–18:00 (Europe/Moscow), окно 14 дней. В окне
всегда есть будни, поэтому свободные слоты гарантированно присутствуют даже если
прогон запущен в выходной.

## Запуск

```bash
cd frontend
npm run test:e2e          # прогон всех сценариев (headless)
npm run test:e2e:ui       # интерактивный UI-режим Playwright
npm run test:e2e:report   # открыть HTML-отчёт последнего прогона
```

## Пользовательские сценарии

### 1. Основное бронирование (happy path) — `booking.spec.ts`

**Предусловие:** есть хотя бы один тип события со свободными слотами.

**Шаги:**
1. Открыть `/`. Виден заголовок «Выберите тип встречи».
2. Нажать «Записаться» у первого типа события.
3. На странице типа виден заголовок «Свободные слоты (14 дней)».
4. Нажать кнопку первого свободного слота (время `HH:mm`).
5. В модалке «Подтверждение записи» заполнить «Ваше имя» и «Email».
6. Нажать «Забронировать».

**Ожидаемый результат:** переход на `/bookings/:id`, виден заголовок
«Вы записаны!», указан email гостя и кнопка «На главную».

### 2. Валидация формы на клиенте — `validation.spec.ts`

**Предусловие:** открыта модалка бронирования для свободного слота.

**Шаги и ожидания:**
- Отправка с пустым именем → ошибка «Укажите имя», модалка не закрывается.
- Ввод некорректного email (`не-email`) → ошибка «Некорректный email».
- Запрос на создание брони на сервер не уходит (URL остаётся на странице типа).

### 3. Конфликт времени, 409 — `conflict.spec.ts`

**Предусловие:** глобальное правило занятости — на одно время не может быть двух
броней (даже для разных типов событий).

**Шаги:**
1. Открыть страницу типа события, открыть модалку для первого свободного слота
   (UI «держит» этот слот).
2. Параллельно занять этот же слот через API (имитация второго гостя).
3. Отправить форму в UI.

**Ожидаемый результат:** уведомление «Время уже занято», модалка закрывается,
переход на страницу подтверждения не происходит.

### 4. Запись видна владельцу — `admin-sees-booking.spec.ts`

**Предусловие:** гость только что создал бронь с уникальным email.

**Шаги:**
1. Пройти основной сценарий бронирования с уникальным email.
2. Перейти в раздел «Владельцу» → вкладка «Записи» (`/admin/bookings`).

**Ожидаемый результат:** в таблице «Предстоящие записи» есть строка с именем и
email только что созданного гостя.
